# Incident Report: Codex/Godex GPT-5.4 VibeProxy Failure

## Date

2026-04-02

## Symptom

Godex/Codex stopped working with GPT-5.4 through the local VibeProxy setup and returned:

```text
ERROR: Bad Request
HTTP error: 400 Bad Request, url: ws://127.0.0.1:8328/v1/responses
```

## What Was Still Healthy

The proxy services themselves were up and advertising GPT models:

- `127.0.0.1:8317` listening
- `127.0.0.1:8328` listening
- `/v1/models` returned `gpt-5.4`
- `/v1/models` returned `gpt-5.4-mini`

Direct HTTP requests to the local Responses endpoint also worked.

That proved the models and the proxy were not completely down.

## Investigation

### 1. Reproduced the real failure path

Running:

```bash
codex exec --skip-git-repo-check -C "/Users/aejjusingh" "Reply with OK only."
```

showed that Codex was trying to connect over websocket:

```text
failed to connect to websocket: HTTP error: 400 Bad Request, url: ws://127.0.0.1:8328/v1/responses
```

### 2. Verified the local shim behavior

The local shim handled regular HTTP Responses requests correctly, but it did not implement websocket upgrade support.

### 3. Isolated the mismatch

The built-in `openai` provider behavior in Codex had shifted toward websocket transport for Responses API sessions.

The local proxy stack supported:

- `POST /v1/responses`
- `GET /v1/models`

but not:

- websocket upgrade on `/v1/responses`

## Root Cause

The break was caused by a transport mismatch:

- Codex used websocket transport for the built-in provider
- the local proxy only supported HTTP

This was not a GPT-5.4 model outage and not a proxy catalog failure.

## Fix Applied

The working fix was to define a custom provider in `~/.codex/config.toml` and disable websocket support explicitly:

```toml
model_provider = "localproxy"

[model_providers.localproxy]
name = "Local Proxy"
base_url = "http://127.0.0.1:8328/v1"
wire_api = "responses"
supports_websockets = false
experimental_bearer_token = "<your-local-proxy-token>"
```

## Why This Fix Works

By moving away from the built-in `openai` provider and into a custom provider, Codex respects:

```toml
supports_websockets = false
```

That forces the client back onto the normal HTTP request path that the proxy actually supports.

## Verification Evidence

These commands succeeded after the config change:

```bash
codex exec --skip-git-repo-check -C "/Users/aejjusingh" "Reply with OK only."
codex exec --skip-git-repo-check -C "/Users/aejjusingh" -m gpt-5.4-mini "Reply with MINI only."
```

Observed outputs:

- `OK`
- `MINI`

## Final Outcome

The GPT-5.4 and GPT-5.4-mini flows were restored without changing the proxy implementation itself.
